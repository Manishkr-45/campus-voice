\chapter*{Abstract}
CampusVoice is a web application built with Flask (Python) and MySQL to collect, manage, and act on student feedback and complaints. The system provides secure user authentication, role-based access (students, staff, admin), complaint submission with attachments, an admin dashboard for review and action, and a lightweight REST-style API for integrations. This report documents the architecture, implementation, database schema, and usage instructions.

\chapter{Introduction}
CampusVoice is designed to simplify how students submit feedback and complaints and how campus staff and administrators manage them. Core goals: shorten response time, centralize records, allow anonymous or attributed reports, and provide analytics to identify recurring issues.

\section{Objectives}
\begin{itemize}
  \item Provide secure login and role-based access
  \item Allow complaints and feedback submission with attachments
  \item Provide an admin interface to review, respond and close items
  \item Persist data in MySQL and expose basic API endpoints
\end{itemize}

\chapter{System Overview}
\section{Technology Stack}
\begin{itemize}
  \item Backend: Flask (Python)
  \item Database: MySQL (mysqlclient + Flask-MySQLdb)
  \item Auth: Flask-Login, Flask-Bcrypt
  \item Frontend: HTML/CSS/JavaScript (static files in `static/`)
\end{itemize}

\section{High-level Architecture}
The application follows a modular Flask blueprint structure: `routes/auth.py`, `routes/main.py`, `routes/complaints.py`, `routes/feedback.py`, and `routes/admin.py`. Extensions are initialized in `app.py` and the database connection uses `flask_mysqldb`.

\chapter{Backend Implementation}
\section{Application Structure}
Key files:
\begin{itemize}
  \item `app.py` — factory creating the Flask app and initializing extensions
  \item `models.py` — `User` model and `user_loader` integration
  \item `routes/*.py` — blueprints for auth, main, complaints, feedback, admin
  \item `schema.sql` — SQL to create database tables
\end{itemize}

\section{Important Endpoints}
\begin{itemize}
  \item `POST /api/auth/login` — login
  \item `POST /api/auth/register` — register
  \item `GET /api/auth/me` — current user
  \item `POST /api/complaints` — submit complaint
  \item `GET /api/complaints` — list complaints (admin)
  \item `POST /api/feedback` — submit feedback
\end{itemize}

\chapter{Database Design}
\section{Main Tables}
\begin{description}
  \item[users] id, username, password_hash, name, email, role
  \item[complaints] id, user_id, title, body, status, created_at, attachment_path
  \item[feedback] id, user_id, message, created_at
\end{description}

Example `schema.sql` snippets (already present in the repository): ensure the `users`, `complaints`, and `feedback` tables exist and that `users.id` is referenced by foreign keys.

\chapter{Methodology}
\section{Development Approach}
Agile-style iterative development: build core auth and data persistence first, then add UI, file uploads, and admin features. Emphasized small, testable units and simple REST patterns.

\chapter{Results}

\begin{itemize}

  \item Complaint submission with file upload persisted to `static/uploads/`
  \item Admin endpoints for review and state changes
